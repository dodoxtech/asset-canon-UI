# Art Direction

> The visual spec — palette, typography, pixel rules, and how every asset is
> produced. This is the **single source of truth** for color and type; other docs
> link here instead of restating values.
>
> **Machine-readable form:** the asset-canon pipeline reads this as
> [`../style-profile.yaml`](../style-profile.yaml) — the shared style brief every
> generation inherits (`prompt_suffix`, palette, `negative`, chroma plate). Edit
> this doc, then mirror the change into that profile.

## Visual target

Late-era **GBA aesthetic** (think *Zelda: The Minish Cap*, *Golden Sun*, *Pokémon
Emerald*): a 240×160-derived virtual resolution, chunky readable pixels, soft
ambient gradients done in *dithering* rather than smooth blends, warm interior
lighting, and crisp UI windows with 1px borders. Cozy workshop, not gritty.

The page leans into asset-canon's own brand colors — **chroma-green `#00B140`** and
**chroma-magenta `#FF00FF`** (the literal chroma-key plates the skill uses) become
the hero accent and secondary accent. A quiet in-joke for anyone who reads the docs.

## Color palette

One palette, 16 colors, used by **every** asset (the same "one palette per batch"
discipline asset-canon enforces). Hex is authoritative; names are for reference.

### Core / environment

| Token | Hex | Use |
|---|---|---|
| `ink` | `#0B0E1A` | Deepest shadow, outlines, letterbox bars |
| `night` | `#141A30` | Page/stage background |
| `panel-dark` | `#1E2A44` | Dialogue window fill, room far-walls |
| `panel-mid` | `#324063` | Floors, mid surfaces |
| `stone` | `#5B6B96` | Mid-tone props, inactive UI |
| `border` | `#8A9AC8` | Window borders, light edges |
| `paper` | `#E8ECF8` | Primary text, UI highlights |

### Accents (brand + signal)

| Token | Hex | Use |
|---|---|---|
| `canon-green` | `#00B140` | **Primary** — brand, the Canon Shard, primary CTA button |
| `green-lite` | `#3CE07A` | Green highlight / glow ramp |
| `green-dark` | `#157A3A` | Green shadow |
| `canon-magenta` | `#FF2CC0` | **Secondary** — sprite accents, the Rune, hover states |
| `gold` | `#FFC83C` | Collectibles, coins, the Key, "+1" pops |
| `gold-dark` | `#C8821E` | Gold shadow |

> Ramps: each accent ships a 3-step ramp (dark → base → lite) so sprites shade
> against the single palette without introducing new hues.

### CRT / atmosphere overlay (CSS only, not in the sprite palette)

- Subtle scanlines at ~8% opacity, a faint vignette toward `ink`, and an optional
  1–2% RGB-split on the stage edge. Toggleable; off under `prefers-reduced-motion`.

## Typography

All type is pixel/bitmap, loaded from Google Fonts (self-host for performance).

| Role | Font | Notes |
|---|---|---|
| Display / title / H1–H2 | **Press Start 2P** | The iconic 8-bit face. Use only at large sizes and short strings — it's heavy at body size. |
| UI labels / HUD / buttons | **Silkscreen** | Tighter, legible at small sizes for counters, menu items, tags. |
| Body / dialogue text | **Pixelify Sans** | Readable pixel face for paragraphs in dialogue windows. |
| Terminal / code / install cmds | **VT323** | Monospace bitmap for the `npx …` / `/plugin …` blocks. |

**Type rules**

- Size in integer multiples of the base grid (8px). Never sub-pixel.
- `font-smooth: never` / `-webkit-font-smoothing: none` so glyphs stay crisp.
- Line length in dialogue windows ≤ 38 characters — GBA textbox cadence.
- Display headings: `letter-spacing` 0; Press Start 2P is already wide.

## Pixel & layout rules

- **Virtual resolution:** design on a **480×270** stage (16:9, a clean 2× of a
  GBA-ish 240×135). Everything is authored at 1× and scaled by **integer factors
  only** (2×, 3×, 4×) to fit the viewport. See
  [interaction-and-motion.md](interaction-and-motion.md#the-virtual-stage).
- `image-rendering: pixelated` on every sprite, canvas, and scaled image.
- **Grid:** 16×16 px tiles for the world; 8px UI spacing grid. Character ≈ 16×24.
- **Outlines:** 1px `ink` selective outline on sprites (not a full black key line)
  for the softer Minish-Cap look.
- **Depth via dithering**, not gradients — Bayer 2×2/4×4 patterns between palette
  steps for skies, glows, and ambient occlusion.

## Asset production plan

The whole point: **every asset is generated by asset-canon**, proving the product.
Map each asset class to the skill that makes it. For the **full counted list of
every individual asset** (slugs, exact sizes, frame counts, and a generation-ready
description of each), see [asset-manifest.md](asset-manifest.md).

| Asset | Skill / command | Spec |
|---|---|---|
| Hero character (idle, walk, pickup) | `asset-sprite` | 16×24, frame-grid spritesheet + atlas; idle 2f, walk 4f, pickup 2f |
| Collectible shards / key / coins | `asset-sprite` | 16×16, 4f sparkle loop each, on chroma-green plate → keyed to alpha |
| World tiles (floor, wall, props) | `asset-sprite` | 16×16 tileset, seamless edges |
| Room backgrounds / vignettes | `asset-illustration` | One per room, drawn against the shared palette |
| Tileable wall/floor textures | `asset-texture` | Seamless 2×2 seam-checked, for large surfaces |
| HUD icons, sound/map buttons, favicon | `asset-icon` | 16×16 + size ladder; favicon family |
| OG / social share card | `asset-social` | Exact OG dimensions, "Play CANON QUEST" framing |
| Early concept sketches only | ChatGPT | Mood/thumbnail exploration before locking the style |

**Pipeline reminder:** lock one style with `/asset-new`, then `/asset-gen` per
batch, `/asset-variants` for states (e.g. shard colors), `/asset-optimize` to pack
spritesheets and export webp/png. Each shipped asset writes its `<slug>.yaml`
descriptor into [`../assets/`](../assets/) so the frontend can place it without
opening the image. This is the same loop a real asset-canon user would run.
